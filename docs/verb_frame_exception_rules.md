# Verb-Frame Exception Rules

## Status

Active specification.

This document defines how the project handles small, fixed exceptions in the verb-frame generation system.

The default rule should remain simple and stable.

Exceptions must be explicit, limited, and data-driven.

## Core Principle

Do not break the main frame architecture for a small number of irregular phrases.

Use:

```text
regular frame + explicit exception frame
```

Example:

```text
go_to + the park → go to the park
go_to + school → go to school
go_home + home → go home
```

Do not allow:

```text
go_to + home → go to home
```

## Why Exceptions Are Needed

Some English phrases do not follow the regular surface pattern.

Examples:

```text
go home
go downtown
go abroad
go upstairs
```

These should not force the whole `go_to` frame to change.

They should be handled as explicit exception frames.

## Exception Item Schema

Recommended schema:

```json
{
  "text": "home",
  "level": "A1",
  "scenario": "daily_routine",
  "part_of_speech": "place_phrase",
  "semantic_group": "place_exception",
  "allowed_frames": ["go_home"],
  "blocked_frames": ["go_to"],
  "surface_pattern": "go home",
  "source_note": "Exception: no preposition after go."
}
```

## Rules

1. Keep the regular frame as the default.
2. Do not let exception items enter regular frame slots.
3. Use `blocked_frames` for known invalid combinations.
4. Use explicit exception frames such as:

   * `go_home`
   * `go_downtown`
   * `go_abroad`
5. Do not infer exceptions automatically.
6. Do not use blacklist as the main solution for predictable exceptions.
7. The exception list should stay small and reviewable.

## Regular vs Exception Examples

Allowed:

```text
I go to school.
I go to the park.
I go to the library.
I go home.
```

Blocked:

```text
I go to home.
I go to downtown.
I go to abroad.
```

## Implementation Guidance

For now, exception handling can be data-driven:

* remove exception items from regular frame slots
* add exception items to explicit exception slots
* keep regular patterns unchanged
* add separate patterns only when the exception phrase is needed

Example:

Regular pattern:

```text
I go to {place}.
```

Uses:

```text
daily_routine_go_to_place_A1
```

Exception pattern:

```text
I go {place_exception}.
```

Uses:

```text
daily_routine_go_place_exception_A1
```

## Completion Criteria

Exception handling is correct when:

1. `I go to home.` is not generated.
2. `I go home.` can be generated by an explicit exception pattern.
3. Regular `go_to` patterns still generate:

   * I go to school.
   * I go to the park.
   * I go to the library.
4. The generator does not need broad special-case logic.
5. The exception item is explicit in slot data.
